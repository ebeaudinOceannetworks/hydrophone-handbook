Monitoring & Troubleshooting

Procedure for addressing common failures (e.g. files not streaming: make INSTR ticket and tag Systems (@nvops) to request power cycle)

create a timeline: validation -> commissioning -> occasional checks -> decommisioning

Validation

data validation: - anything before the deployment, while the device is still at MTC - tank test - extra ocean sonics validation test (piston)

Commissioning

confluence page: https://internal.oceannetworks.ca/pages/resumedraft.action?draftId=272761007&draftShareId=20e85281-232e-4b9e-b2ac-fbe56256798a& data commissioning:

  • once deployed, as it starts streaming the data
  • create a commisioning JIRA DATA ticket (see example for hydrophone)
    • put general info (device name, device ID, device Code, current deployment, device history, calibration date)
  • create a commisioning annotation, then close once commisioning is done
    • check if there are other open annotations to this device
    • check sampling rate
    • no need for primary sensor for hydrophones
  • check data stream in oceans 3.0
    • over a few months of data
    • if possible, compare incoming spectrogram to past devices at same location (compare sensitivities/noise levels)
  • device action in oceans 3.0
  • close commisioning annotation
  • assign a reviewer in JIRA (the other DAQ IPP), they will review and close the ticket. reivew takes ~30 min. (there is a confluence page)

List of common issues, potential root causes, and how to report it:

  • No data stream -> Junction Box down, network down Looking at the data availability plot in the Hydrophone Dashboard under an individual hydrophone, if there is incoming data missing for .flac/.wav, this needs to be report as an INSTR ticket. If the missing data is the derived .mat file instead, this could be related to somethign else, check next issue.

  • Audio data archived but no .mat/.png -> check with DSO, this might be related to sensitivity curve not updated. This is an internal software system-related problem.

  • Very bad data -> instrument health (temp, humidity, battery), biofouling (very hard to detect, happens over the course of several months), interference that’s masking data significantly (from nearby instruments)

  • clock synchronization (onc Links & icLinks)

How to report an instrument issue

Create a JIRA INSTR ticket, making sure to add device Code, device ID, location in the title with the name of the issue. Using data to assess system problems

  • wav/flac: if while listening something sounds awry, but this might not be obvious if the issue lies outside the range of audible frequencies.
  • spectrogram: useful for detecting clipping, interference, instrument failure (totally black)
  • spd plots: useful for detecting ambient noise, anomalies, power supply interference (electrical noise, mechanical noise)
